# 代码结构规范

> **📎 何时读此文件**: 移动/重命名文件前、重构代码前、修改函数签名前、新增模块时
> **对应原则**: DBG-05（禁止悄悄修改语义）、DBG-06（禁止擅自优化结构）、AI-01（禁止无确认重构）
> **相关文件**: `spec/project-structure.md`, `rules/testing.md`

## 核心原则

> **任何文件移动、重命名、新增顶层模块，必须先征得用户确认。**

---

## 禁止行为

❌ **擅自优化代码结构**：
- 不得将函数"觉得应该"移到 utils
- 不得将多个模块的公共逻辑抽成新文件而不通知
- 不得重命名已有文件
- 不得删除"看起来没用"的代码

❌ **引入循环导入**：
- `utils/` 不得导入 `services/`、`core/`、`app/`
- `models/` 不得导入 `services/`、`core/`
- `schemas/` 不得导入 `models/`

❌ **破坏已有接口**：
- 不得修改已有函数的返回值语义
- 不得修改参数类型/顺序
- 不得删除已有函数
- 不得修改异常抛出方式

---

## 变更流程

### 文件移动

1. **输出计划**：
   ```
   我计划将 backend/core/orchestrator.py 中的 _auto_generate_pdf 方法
   移动到 backend/utils/pdf_generator.py 中，因为：
   1. 该方法与 PDF 生成相关，不属于编排逻辑
   2. 其他模块可能也需要 PDF 生成功能
   3. 当前位置导致 core 模块过于臃肿
   
   影响范围：
   - orchestrator.py 需要更新 import
   - 需要验证所有调用点
   
   请确认是否执行？
   ```

2. **用户确认**：等待用户批准

3. **执行移动**：
   - 创建新文件
   - 复制代码
   - 更新旧文件引用
   - 删除旧代码

4. **验证编译**：
   - 运行测试
   - 检查导入
   - 验证功能

5. **记录变更**：更新 `spec/project-structure.md`

## 函数签名保持

修改函数时，必须保持：

| 元素 | 是否可修改 | 要求 |
|------|-----------|------|
| 函数名 | ❌ 不可 | 保持原名 |
| 参数名 | ❌ 不可 | 保持原名 |
| 参数顺序 | ❌ 不可 | 保持原顺序 |
| 参数类型 | ❌ 不可 | 保持原类型 |
| 参数默认值 | ⚠️ 谨慎 | 新增可选参数可以 |
| 返回值类型 | ❌ 不可 | 保持原类型 |
| 返回值语义 | ❌ 不可 | 空列表 vs None |
| 异常类型 | ❌ 不可 | 保持原异常 |

**必须修改时**：
- 标注 `@breaking-change`
- 通知所有调用方
- 提供迁移方案
- 更新文档

## 重构规范

### 重构传染范围硬约束

> **任何单次修改超过 3 个文件的，必须先输出影响分析并获得用户确认。超过 5 个文件的修改必须拆分为多个独立步骤。**

| 影响文件数 | 要求 |
|-----------|------|
| 1-2 个文件 | 可直接修改，完成后自审查 |
| 3-5 个文件 | 必须先输出影响分析（哪些文件、为什么、是否破坏已有功能），获得确认后执行 |
| 6+ 个文件 | 必须拆分为多个独立步骤，每步不超过 5 个文件，逐步确认 |

**影响分析模板**（3+ 文件时必须输出）：

```markdown
## 重构影响分析

**变更原因**: [为什么要做这个重构]
**影响文件数**: [N] 个

### 文件变更清单

| 文件 | 变更类型 | 变更内容 | 是否破坏兼容 |
|------|----------|----------|-------------|
| `path/to/file1.py` | 修改签名 | 添加可选参数 `timeout` | ❌ 向后兼容 |
| `path/to/file2.py` | 更新调用 | 适配新签名 | ❌ |
| `path/to/file3.py` | 移动函数 | 从 core 移到 utils | ⚠️ 需要更新 import |

### 风险评估

- 循环导入风险: [低/中/高]
- 测试破坏风险: [低/中/高]
- 功能回归风险: [低/中/高]

### 回滚方案

[如果出问题，如何回滚]

---

请确认是否执行？
```

### 允许的重构

- ✅ 提取重复代码为函数（保持语义）
- ✅ 优化算法（保持输入输出）
- ✅ 添加类型注解（不改变逻辑）
- ✅ 添加注释和文档（不改变逻辑）

### 禁止的重构

- ❌ 修改函数签名
- ❌ 修改类继承关系
- ❌ 修改数据库表结构
- ❌ 修改 API 响应格式
- ❌ 删除"看起来没用"的代码
- ❌ 修改配置项含义
- ❌ 单次修改超过 5 个文件（必须先拆分）
- ❌ 在无影响分析的情况下修改超过 2 个文件

## 代码审查清单

AI 修改代码后，必须自检：

- [ ] 是否修改了已有函数签名
- [ ] 是否修改了返回值语义
- [ ] 是否引入循环导入
- [ ] 是否破坏已有测试
- [ ] 是否删除"看起来没用"的代码
- [ ] 是否移动了文件
- [ ] 是否重命名了模块
- [ ] 是否更新了所有引用

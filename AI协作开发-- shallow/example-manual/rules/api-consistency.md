# 接口一致性规范

> **📎 何时读此文件**: 修改 API 接口时、前后端联调异常时、字段命名不明确时
> **对应原则**: AI-02（禁止前端/后端只改一端）
> **相关文件**: `spec/api-contract.md`, `rules/testing.md`

## 核心原则

> **涉及接口变更的任务，必须同时输出前端和后端的修改计划，并保证字段名、类型、错误码完全一致。**

---

## 禁止行为

❌ **只改前端或只改后端**：
- 修改了 API 返回字段，但只更新了后端，前端还按旧字段解析
- 修改了前端发送格式，但后端还按旧格式解析

❌ **字段不一致**：
- 前端传 `user_id`，后端收 `userId`
- 前端传字符串，后端期望整数
- 前端传数组，后端期望对象

❌ **错误码不一致**：
- 后端返回 400，前端判断 422
- 后端返回 `{"error": "..."}`，前端判断 `{"message": "..."}`

---

## 变更流程

### 1. 输出变更计划

```markdown
## 接口变更计划

### 变更接口
POST /api/missions

### 变更内容
- 新增字段: `squad_config` (原 `squad` 的别名)
- 字段类型: JSON Object
- 是否必填: 否

### 前端修改
- [ ] 发送请求时使用 `squad_config` 字段
- [ ] 兼容旧 `squad` 字段（向后兼容）

### 后端修改
- [ ] 接收 `squad_config` 字段
- [ ] 兼容旧 `squad` 字段
- [ ] 更新 API 文档

### 验证
- [ ] curl 测试新接口
- [ ] curl 测试旧接口（向后兼容）
- [ ] 前端联调
```

### 2. 执行修改

**后端先行**：
1. 后端实现新接口
2. 后端保持向后兼容
3. 后端验证通过

**前端跟进**：
1. 前端更新 API 调用
2. 前端处理新字段
3. 前端联调测试

### 3. 验证

```bash
# 验证新接口
curl -X POST http://localhost:8000/api/missions \
  -H "Content-Type: application/json" \
  -d '{"title": "test", "squad_config": {"agents": []}}'

# 验证旧接口（向后兼容）
curl -X POST http://localhost:8000/api/missions \
  -H "Content-Type: application/json" \
  -d '{"title": "test", "squad": {"agents": []}}'
```

## 向后兼容

### 原则

- 新增字段：可选，不影响旧代码
- 修改字段：不修改，而是新增字段
- 删除字段：标记为废弃，保留一段时间
- 修改语义：新增接口，保留旧接口

### 废弃字段处理

```python
# 后端接收
squad_config = mission_in.get("squad_config") or mission_in.get("squad", {})

# 前端发送
payload = {
    "squad_config": {...}  # 新字段
    # "squad": {...}  # 旧字段，废弃
}
```

## 文档更新

接口变更后必须更新：

1. **接口文档**（`spec/api-contract.md`）
2. **代码注释**（函数 docstring）
3. **变更日志**（`spec/api-contract.md` 中的变更日志）

---

> ⚠️ **重要**：只改一端是前后端联调中最常见的 bug 来源。严格执行此规范。

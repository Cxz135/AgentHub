# 完成定义 (Definition of Done)

> **📎 何时读此文件**: 标记任务完成前、提交 PR 前、自审查时
> **对应原则**: DEV-06（彻底打通才算完成）、AI-04（禁止为通过测试而降级）
> **相关文件**: `rules/testing.md`, `spec/user-journey.md`

## 核心原则

> **某一链路彻底打通，该任务才算完成。AI 不得因为一个问题迟迟不解决，或者为了通过测试去把之前敲定的方法降级以通过。**

---

## 完成标准

### 功能完成

- [ ] 功能实现符合需求
- [ ] 输入输出符合预期
- [ ] 边界情况处理
- [ ] 异常情况处理
- [ ] 性能满足要求

### 代码质量

- [ ] 代码风格一致
- [ ] 函数职责单一
- [ ] 有类型注解
- [ ] 有文档字符串
- [ ] 有日志记录
- [ ] 无重复代码
- [ ] 无循环导入

### 测试完成

- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 边界测试通过
- [ ] 测试覆盖率达到 80%（核心模块）

### 文档完成

- [ ] 代码注释完整
- [ ] 接口文档更新
- [ ] 变更日志记录
- [ ] 踩坑日志记录（如有）

### 链路打通

- [ ] 前端 → 后端 → 数据库 → 返回 → 前端
- [ ] 用户输入 → 处理 → 产物 → 下载/预览
- [ ] 异常输入 → 错误处理 → 用户提示

---

## 禁止行为

❌ **为了完成而降级**:
```python
# 错误：为了通过测试而修改断言
assert result == "expected"  # 改为 assert result is not None

# 错误：为了完成而跳过验证
if True:  # 原为 if validate(data)
    process(data)

# 错误：为了完成而删除测试
# os.remove("test_failed.py")
```

❌ **为了完成而绕过**:
```python
# 错误：为了完成而忽略异常
try:
    process()
except:
    pass  # 忽略所有异常

# 错误：为了完成而返回硬编码
if not result:
    return "成功"  # 硬编码
```

❌ **为了完成而修改已有功能**:
```python
# 错误：为了完成新功能而破坏旧功能
def old_function():
    # 原有逻辑
    return old_result
    # 新增：为了新功能而修改
    return new_result  # 破坏原有功能
```

## 验收流程

1. **功能演示**：
   - 用户输入 → 系统处理 → 产物展示
   - 录制或截图记录

2. **日志检查**：
   - 查看完整日志链路
   - 确认无错误、无异常
   - 确认耗时合理

3. **测试验证**：
   - 运行所有测试
   - 检查覆盖率
   - 确认无失败

4. **链路验证**：
   - 前端操作 → 后端处理 → 数据库 → 返回
   - 确认每个环节正常

5. **异常验证**：
   - 输入错误数据
   - 模拟网络中断
   - 确认错误处理正常

## 完成声明

任务完成后，AI 必须输出：

```
✅ 任务完成

功能: [功能名称]
完成标准:
- [x] 功能实现
- [x] 测试通过
- [x] 链路打通
- [x] 日志正常
- [x] 文档更新

验证结果:
- 输入: [输入示例]
- 输出: [输出示例]
- 耗时: [耗时]
- 日志: [关键日志]

已知问题:
- [ ] 无

下一步:
- [ ] 无
```

## 未完成任务处理

如果无法完成：

1. **记录原因**：
   - 为什么不能完成
   - 尝试了什么方案
   - 遇到了什么障碍

2. **记录进展**：
   - 完成了什么
   - 还有什么没完成
   - 阻塞点在哪里

3. **更新 todo**：
   - 将未完成任务移回 todo.md
   - 标记为阻塞状态
   - 记录原因

4. **用户确认**：
   - 向用户说明情况
   - 提供解决方案
   - 等待用户决策

---

> ⚠️ **重要**：不完整的任务比不完成的任务更糟糕。宁可明确说"没完成"，也不要说"完成了"但有问题。

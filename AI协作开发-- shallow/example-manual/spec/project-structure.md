# 项目目录结构规范

> **📎 何时读此文件**: 新增模块/目录时、移动文件前、怀疑循环导入时
> **对应原则**: PRJ-04（目录职责解耦）、DBG-06（禁止擅自优化结构）

## 核心原则

> **每个目录必须有且仅有一个明确的职责。新增功能不得随意放入已有目录。**

---

## 目录职责矩阵

| 目录 | 职责 | 允许放入 | 禁止放入 |
|------|------|----------|----------|
| `app/api/` | HTTP 路由定义 | 路由处理器、依赖注入 | 业务逻辑、数据库操作 |
| `app/models/` | 数据库模型 | SQLAlchemy 模型 | 业务逻辑、API 定义 |
| `app/schemas/` | Pydantic 模型 | 请求/响应模型 | 业务逻辑、数据库模型 |
| `core/` | 核心业务逻辑 | 编排器、工作流 | 路由、数据库模型 |
| `services/` | 业务服务层 | 业务方法、事务控制 | 路由、工具函数 |
| `utils/` | 通用工具 | 纯函数、文件转换 | 业务逻辑、路由 |
| `db/` | 数据库配置 | 连接、迁移脚本 | 业务逻辑、模型 |
| `config/` | 配置管理 | YAML、环境变量 | 业务逻辑、工具 |
| `agents/` | Agent 定义 | Agent 类、适配器 | 业务逻辑、路由 |
| `llm/` | LLM 后端 | 模型调用、后端管理 | 业务逻辑、Agent 定义 |
| `workflows/` | 工作流定义 | 工作流类、图构建 | 业务逻辑、路由 |

---

## 新增目录流程

如果需要新增目录，必须满足以下条件：

1. **不重复**：现有目录无法容纳该功能
2. **单一职责**：新目录有明确的、唯一的职责
3. **有规范**：必须创建 README.md 说明目录职责
4. **经确认**：必须经过技术负责人确认

---

## 文件移动规范

任何文件移动、重命名、新增顶层模块，必须遵循以下流程：

1. **输出计划**：AI 必须输出 "我计划将 XXX 移到 YYY，因为 ZZZ"
2. **用户确认**：等待用户批准
3. **执行移动**：执行文件操作
4. **更新引用**：更新所有引用路径
5. **验证编译**：确保没有循环导入或引用错误
6. **记录变更**：更新 `spec/project-structure.md`

**禁止**：AI 擅自移动文件、重命名模块、提取公共逻辑到新文件。

---

## 循环导入预防

- `utils/` 不得导入 `services/`、`core/`、`app/`
- `models/` 不得导入 `services/`、`core/`
- `schemas/` 不得导入 `models/`
- `config/` 只能被其他目录导入，不得导入其他目录

# 代码审查技能

## 技能定义

**名称**: `code-review`
**描述**: 系统化代码审查技能，要求 AI 按多维度检查代码变更，输出结构化的审查意见
**触发条件**: 用户提交 PR、请求审查变更、或代码修改完成后自检
**何时读此文件**: 当用户说"review 一下"、"帮我看看这段代码"、"检查 PR"、或在完成任务后进行自检时

---

## 审查维度

每个变更必须从以下 5 个维度进行审查。每个 finding 必须标注对应维度：

| 维度 | 检查要点 | 对应规则文件 |
|------|----------|-------------|
| **正确性 (correctness)** | 逻辑是否正确、边界情况是否处理、是否会引入 bug | — |
| **安全性 (security)** | 是否有注入风险、敏感数据是否暴露、权限检查是否完备 | — |
| **性能 (performance)** | 是否有不必要的循环、N+1 查询、阻塞调用 | `rules/logging.md`（调用计时） |
| **可维护性 (maintainability)** | 代码是否清晰、日志是否完整、是否易于调试 | `rules/logging.md`, `rules/code-structure.md` |
| **规范符合度 (compliance)** | 是否违反任何已定义的规则 | `rules/*.md`（全部规则文件） |

---

## 严重等级

| 等级 | 标签 | 含义 | 是否阻塞合并 |
|------|------|------|-------------|
| 🔴 Blocker | 必须修复 | 会导致功能异常、安全漏洞、数据丢失 | ✅ 阻塞 |
| 🟡 Major | 应该修复 | 可能导致边界情况异常、违反核心规范 | ✅ 建议阻塞 |
| 🔵 Minor | 建议修复 | 代码风格、命名、可读性问题 | ❌ 不阻塞 |
| ⚪ Nit | 锦上添花 | 可选的优化建议 | ❌ 不阻塞 |

## 审查流程

### 1. 获取变更内容

```bash
# 查看当前分支相对于 main 的变更
git diff main...HEAD

# 或查看已暂存但未提交的变更
git diff --cached

# 或查看指定 commit
git show <commit-hash>
```

### 2. 逐文件审查

对每个变更文件：

1. **通读变更**：理解整体意图
2. **逐段检查**：按 5 个维度逐段检查
3. **交叉验证**：检查变更是否影响了其他文件的调用方
4. **规则对比**：对照 `rules/` 中的规则检查违规项

### 3. 输出审查意见

## 审查检查清单（快速版）

当时间有限时，至少检查以下 10 项：

- [ ] **函数签名**：是否修改了已有函数签名？（`rules/code-structure.md`）
- [ ] **前后端一致**：接口变更是否两端都改了？（`rules/api-consistency.md`）
- [ ] **硬编码**：是否新增了写死逻辑？（`rules/ai-decision.md`）
- [ ] **日志**：新增函数是否有入口/分支/出口/错误日志？（`rules/logging.md`）
- [ ] **异常处理**：except 块是否记录了完整异常？（`rules/logging.md`）
- [ ] **测试**：是否有对应测试且没有修改断言？（`rules/testing.md`）
- [ ] **临时代码**：是否有 `print()` 或未标记的调试代码？（`skill/debugging.md`）
- [ ] **文件移动**：是否移动/重命名了文件？（`rules/code-structure.md`）
- [ ] **循环导入**：是否引入了循环导入？（`spec/project-structure.md`）
- [ ] **语义变更**：返回值语义、异常类型是否变化？（`rules/code-structure.md`）

## 自审查模式

AI 在完成任何代码修改后，必须执行快速自审查：

```
🤖 自审查完成：

变更文件: [N] 个
检查结果:
- [x] 无函数签名变更
- [x] 无前后端不一致
- [x] 无新增硬编码
- [x] 日志完整
- [x] 测试通过
- [ ] ⚠️ 发现 1 处 #TEMP_DEBUG# 未清理 → 已清理

结论: ✅ 通过自审查 / ⚠️ 有 [N] 项待处理
```

## 与规则体系的映射

审查时必须对照以下规则文件（按 `readme.md` 中的 AI Agent 路由表按需加载）：

| 规则文件 | 关键检查点 | 规则 ID |
|----------|-----------|---------|
| `rules/logging.md` | 日志完整、错误不写死、调用计时 | DEV-01, DEV-02, DEV-05 |
| `rules/testing.md` | 测试不绕过、断言不修改 | DBG-04, AI-04 |
| `rules/ai-decision.md` | 不用写死逻辑、模糊任务先提问 | DEV-03, AI-05 |
| `rules/code-structure.md` | 不修改签名、不擅自移动文件、重构不超范围 | DBG-05, DBG-06, AI-01 |
| `rules/api-consistency.md` | 前后端同时改、字段一致 | AI-02 |
| `rules/definition-of-done.md` | 全链路打通、不降级完成 | DEV-06, AI-04 |

---

> 💡 **提示**：审查不是找茬。好的审查应同时指出做得好的地方和需要改进的地方。

# 面试准备：数据库与数据模型

## Q0: 请简单介绍一下 AgentHub 的数据库设计和数据模型

**参考答复：**

AgentHub 使用 **SQLAlchemy ORM + SQLite** 作为数据持久化方案，包含两个独立的 SQLite 数据库：

**主数据库（agenthub.db）**：存储所有业务数据，包含 9 张核心表：
- `users` — 用户账号和认证信息
- `conversations` — 对话会话（支持单人和群聊模式）
- `messages` — 对话消息（JSON 格式内容 + mentions 关系）
- `custom_agents` — 用户自定义的 Agent 配置
- `skills` — Skill 市场（发布、安装、Fork）
- `skill_installs` — 用户安装 Skill 的关联表
- `memory_entries` — 长期记忆条目
- `user_profiles` — 用户画像
- `artifacts` — Agent 产出的结构化产物

**记忆数据库（agenthub_memory.sqlite）**：专用于 LangGraph 的 checkpointer（`AsyncSqliteSaver`），存储对话状态图的检查点数据。

**设计原则**：
- SQLAlchemy 声明式模型（Declarative Base）——模型类即表定义
- 使用原始 SQL 查询避免 relationship 初始化问题
- 启动时自动执行 schema 迁移（ALTER TABLE 添加缺失列）
- 通过 `SessionLocal` 工厂管理 session 生命周期

## Q1: 为什么选择 SQLite 而不是 PostgreSQL？有什么权衡？

**参考答复：**

选择 SQLite 的原因和权衡：

**优势**：
- **零部署**：不需要安装和配置数据库服务，`pip install` + 一个文件即可
- **零运维**：不需要管理数据库进程、备份策略、连接池
- **单机性能**：对于中小规模（几万条数据），SQLite 的读写性能完全够用
- **便携性**：整个数据库就是一个文件，可以轻松备份和迁移
- **适合原型**：快速迭代阶段，不需要为数据库运维分心

**权衡**：
- **并发写入**：SQLite 只支持单写入者（WAL 模式下可并发读），高并发写入场景会成为瓶颈
- **网络访问**：不支持远程连接，只能单机部署
- **功能限制**：缺少 PostgreSQL 的一些高级功能（如全文搜索、数组类型、JSONB 索引等）
- **扩展性**：如果将来需要多实例部署，SQLite 无法共享

**迁移路径**：由于使用了 SQLAlchemy ORM，从 SQLite 迁移到 PostgreSQL 只需要：
1. 修改 `DATABASE_URL` 连接字符串
2. 安装 `psycopg2` 驱动
3. 调整少量 SQLite 特有的 SQL 语法
4. 几乎不需要修改 ORM 模型

## Q2: messages 表的 content 为什么是 JSON 格式？有什么好处？

**参考答复：**

messages 表的 content 使用 JSON 格式而不是纯文本：

```json
{
  "text": "用户的主要消息内容",
  "mentions": ["agent_id_1", "agent_id_2"],
  "attachments": [{"type": "file", "url": "...", "name": "..."}],
  "metadata": {"source": "web", "timestamp": "..."}
}
```

**好处**：
1. **结构化**：一条消息可以包含多种内容（文本 + @mention + 附件 + 元数据），纯文本无法表达
2. **可扩展**：新增字段（如"消息表情反应"）不需要 ALTER TABLE，只需在 JSON 中加字段
3. **前端友好**：前端直接解析 JSON 渲染不同的内容块
4. **查询灵活**：SQLite 支持 JSON 函数（如 `json_extract`），可以按 JSON 字段查询

**代价**：
- 无法对 JSON 内部字段建索引（SQLite 的限制）
- 查询 content 内部字段比普通列慢

## Q3: custom_agents 表中的 memory_config/planning_config/validation_config 是如何存储和使用的？

**参考答复：**

这三个配置以 **JSON 字符串** 形式存储在 `custom_agents` 表的 TEXT 列中：

**存储格式**（以 memory_config 为例）：
```json
{
  "strategy": "sliding_window",
  "window_size": 10,
  "summary_threshold": 4000
}
```

**使用流程**：
1. 从数据库读取 JSON 字符串
2. `json.loads()` 解析为 Python dict
3. 在 Agent 注册时传入配置：
   ```python
   agent = CustomAgent(
       agent_id=...,
       system_prompt=...,
       llm_backend=...,
       validation_config=json.loads(row.validation_config or "{}"),
   )
   ```

**为什么用 JSON 而非独立列**：
- 配置结构可能随版本变化（新增配置项）
- 三个配置的字段数量和类型差异大
- JSON 的灵活性允许不同 Agent 有不同配置

**代价**：不能对配置字段建索引、不能做 SQL 级联查询。

## Q4: 项目的数据库迁移策略是什么？如何在不丢失数据的情况下更新 schema？

**参考答复：**

项目采用**启动时自动迁移**的策略：

在 `app/main.py` 的 lifespan handler 中：
```python
# 使用原始 SQL 检查列是否存在，不存在则 ALTER TABLE
async with engine.begin() as conn:
    # 检查 conversations 表的 is_pinned 列
    result = await conn.execute(text("PRAGMA table_info(conversations)"))
    columns = [row[1] for row in result]
    if "is_pinned" not in columns:
        await conn.execute(text(
            "ALTER TABLE conversations ADD COLUMN is_pinned BOOLEAN DEFAULT FALSE"
        ))
```

**特点**：
- **自动化**：不需要手动运行迁移脚本，启动即迁移
- **非破坏性**：只做 ADD COLUMN，不做 DROP COLUMN（SQLite 不支持）
- **幂等**：检查列是否存在再添加，重复执行安全

**与 Alembic 的对比**：
- Alembic 功能更强大（支持任意 DDL、版本管理、回滚）
- 但当前项目的 schema 变化简单（主要是加列），启动时迁移更轻量
- 如果 schema 变化变得复杂，应该切换到 Alembic

## Q5: skills 表和 skill_installs 表的关系是什么？为什么需要两张表？

**参考答复：**

这是一个**多对多关系**的标准设计：

**skills 表**：Skill 的定义（类似于"App Store 中的应用"）
- slug（唯一标识）、name、code（Skill 的 Markdown 内容）
- author_id、is_published、install_count

**skill_installs 表**：用户-Skill 的安装关系（类似于"用户安装的应用"）
- user_id + skill_id （联合主键）
- installed_at（安装时间）

**为什么需要两张表**：
- 一个 Skill 可以被多个用户安装（1:N）
- 一个用户可以安装多个 Skill（N:1）
- 多对多关系必须通过中间表实现
- 分离定义和关系：Skill 更新不影响已安装用户的记录

**ORM 关系**：
```python
# Skill 模型
installs = relationship("SkillInstall", back_populates="skill")

# SkillInstall 模型
skill = relationship("Skill", back_populates="installs")
user = relationship("User", back_populates="skill_installs")
```

## Q6: UserProfile（用户画像）表是怎么设计的？和 memory_entries 的关系是什么？

**参考答复：**

**UserProfile 表结构**：
```python
class UserProfile(Base):
    user_id = Column(Integer, primary_key=True)
    summary_text = Column(Text)        # 自然语言画像摘要
    traits_json = Column(Text)         # JSON: 用户特征列表
    preferences_json = Column(Text)    # JSON: 用户偏好键值对
    updated_at = Column(DateTime)
```

**与 memory_entries 的关系**：
- memory_entries 是**原始数据**（LLM 提取的结构化事实）
- UserProfile 是**聚合视图**（LLM 将多条记忆总结为画像）

**数据流**：
1. 对话 → MemoryExtractor 提取事实 → 写入 memory_entries
2. 累积到一定量 → UserProfiler 将所有记忆喂给 LLM → 生成画像文本
3. 增量更新：新记忆 → 检查是否改变画像 → 必要时重新生成

**为什么分离**：
- memory_entries 是细粒度的（每条记忆独立），适合语义检索
- UserProfile 是粗粒度的（一次性注入 system prompt），适合快速了解用户
- 两者服务于不同的上下文注入场景

## Q7: Artifact（产物）表的设计有什么考量？为什么单独一张表而不是放在 messages 中？

**参考答复：**

Artifact 独立成表的原因：

**与 Message 的本质区别**：
- Message 是对话记录（谁说了什么）
- Artifact 是对话的产出物（Agent 生成了什么）

**独立成表的好处**：
1. **独立生命周期**：Artifact 可能在对话结束后被单独引用、修改、导出
2. **独立展示**：前端以卡片/面板形式展示 Artifact，而非内嵌在消息流中
3. **类型系统**：Artifact 有明确的 type 枚举（code/html_preview/markdown），方便按类型过滤
4. **查询效率**：查询"这个对话产出了哪些代码"不需要解析 messages 的 JSON content

**表结构**：
```python
class Artifact(Base):
    id, conversation_id, agent_id
    type  # code | html_preview | markdown
    title, content, language
    created_at
```

## Q8: 数据库的 session 管理是怎么做的？如何避免 session 泄漏？

**参考答复：**

Session 管理通过 SQLAlchemy 的 `SessionLocal` 工厂模式：

```python
# db/database.py
engine = create_engine(DATABASE_URL)
SessionLocal = sessionmaker(bind=engine, autocommit=False, autoflush=False)

# 使用方式（FastAPI dependency）
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()  # 确保 session 关闭

# 路由中使用
@router.get("/api/...")
def endpoint(db: Session = Depends(get_db)):
    ...
```

**防泄漏措施**：
1. **依赖注入**：FastAPI 的 `Depends` 确保请求结束后自动调用 `db.close()`
2. **finally 块**：try/finally 保证即使异常也会关闭
3. **Orchestrator 中的手动管理**：Orchestrator 持有的 db_session 引用，在需要时手动创建和关闭

**潜在问题**：Orchestrator 作为单例持有的 session 可能在长时间运行后过期。当前通过定期检查和重建来缓解。

## Q9: conversation 表如何支持单人对话和群聊两种模式？

**参考答复：**

Conversation 表通过 `mode` 和 `participants` 字段区分：

**单人对话**：
- `mode = "single"`
- `participants` 可以为空（只有当前用户和默认 Agent）
- 消息通过 `conversation_id` 关联

**群聊模式**：
- `mode = "group"`
- `participants` 包含参与的 Agent ID 列表（JSON 数组）
- `squad_config` 包含群聊配置（如消息分发策略、汇总方式）

**群聊的消息流**：
1. 用户发送消息 → Orchestrator 检测群聊模式
2. 消息发送给 participants 列表中的所有 Agent
3. 各 Agent 回复后汇总展示

这个设计让同一张表支持两种模式，通过 mode 字段做路由区分，避免了两张表的维护成本。

## Q10: 为什么有些查询使用原始 SQL（`db.execute(text(...))`）而不是 SQLAlchemy ORM？

**参考答复：**

项目中有几处使用了原始 SQL：

```python
# 加载用户 Skills 时
result = db.execute(text(
    "SELECT slug, code, author_name FROM skills WHERE author_id IS NOT NULL AND author_id > 0"
))
```

**原因**：
1. **避免 relationship 初始化问题**：某些 ORM 模型的 relationship 在特定上下文中会报错（如缺少关联对象），原始 SQL 绕过了 ORM 的自动加载
2. **性能**：简单查询不需要 ORM 的对象映射开销
3. **灵活性**：某些查询（如 PRAGMA、JSON 函数）ORM 表达不够直接

**设计原则**：
- 简单查询（SELECT 少量列）→ 原始 SQL
- 复杂对象操作（CRUD） → ORM
- Schema 迁移（ALTER TABLE） → 原始 SQL（ORM 不支持 DDL 迁移）

这不是"ORM vs 原始 SQL"的二选一，而是根据场景选择最合适的工具。

## Q11: memory_entries 表的 decay_factor 和 access_count 是如何更新和维护的？

**参考答复：**

这两个字段是长期记忆衰减机制的核心：

**decay_factor（衰减因子）**：
- 初始值：基于记忆的 importance（重要性越高，初始衰减因子越低）
- 更新：每次 `on_conversation_turn()` 中由 MemoryDecayer 批量更新
- 公式：`new_decay = decay_factor * time_factor * access_penalty`
- 当 decay_factor 降到极低阈值 → 记忆标记为"休眠"

**access_count（访问次数）**：
- 更新时机：记忆被语义检索命中时（`augment_context()` 中每次检索到都 +1）
- 作用：被频繁访问的记忆衰减更慢（boost 机制）
- 实现：`self._decayer.boost(memory_id)` 增加访问计数

**更新方式**：使用 SQLAlchemy 的批量更新，避免逐条更新的性能问题。

## Q12: 如果数据库文件损坏了怎么办？有什么恢复策略？

**参考答复：**

当前项目的数据库恢复策略：

**预防措施**：
- SQLite WAL 模式（Write-Ahead Logging）提高可靠性
- 定期自动 checkpoint

**备份策略**（当前未实现，但建议）：
```bash
# SQLite 在线备份
sqlite3 agenthub.db ".backup agenthub_backup.db"
```

**恢复方案**：
1. ChromaDB（`chroma_db/`）可以重建（从原始文档重新入库）
2. LangGraph checkpointer（`agenthub_memory.sqlite`）丢失只影响当前对话状态，不影响历史消息
3. 主数据库（`agenthub.db`）最关键——包含用户、对话历史、Agent 配置

**当前限制**：没有自动备份机制。对于生产部署，应该：
- 定时备份数据库文件
- 使用 SQLite 的 `.dump` 命令导出 SQL 文本
- 考虑迁移到 PostgreSQL（自带 replication）

## Q13: 数据库查询性能有哪些优化点？

**参考答复：**

当前性能优化措施：

1. **分页查询**：会话列表、消息列表都有分页限制
2. **延迟加载**：relationship 默认 lazy loading，只在需要时加载
3. **索引**：主键和外键有自动索引
4. **批量操作**：记忆衰减使用批量更新而非逐条
5. **JSON 查询**：对于 messages 的 JSON content 查询，使用 SQLite 的 `json_extract()` 函数

**可优化的点**：
- 对高频查询列（如 conversations.user_id, messages.conversation_id）显式建索引
- 对 UserProfile 的最近更新时间做缓存（避免每次对话都查询）
- messages 表增长后考虑分区（按 conversation_id 或时间）
- 对于统计类查询（如"用户总共多少条消息"），使用缓存而非实时 COUNT

SQLite 在万级数据量下性能完全够用，优化需求在百万级数据时才会出现。

## Q14: 数据库连接在并发请求下是否安全？SQLite 的并发限制如何解决？

**参考答复：**

SQLite 的并发特性：
- **读写锁**：WAL 模式下支持一写多读
- **写锁**：同一时间只有一个写入者

**项目中的处理**：
- 每个请求通过 `SessionLocal()` 获取独立 session
- FastAPI 的 async 模型下，多个请求可以并发读取
- 写入操作（如保存消息）在请求处理中顺序执行

**并发瓶颈场景**：
- 多个用户同时创建 Agent → 写入 `custom_agents` 表 → SQLite 写锁排队
- 多个用户同时聊天 → 写入 `messages` 表 → 同上

**解决方案（如果需要高并发写入）**：
1. 设置 SQLite 的 `busy_timeout`（等待而非立即报错）
2. 使用 WAL 模式（已启用）
3. 对于写入密集的操作，考虑批量写入或异步队列
4. 迁移到 PostgreSQL（终极方案）

## Q15: 表之间有哪些外键关系？如何保证数据一致性？

**参考答复：**

核心外键关系：

```
users ──┐
        ├── conversations (user_id)
        ├── messages (user_id/agent_id)
        ├── custom_agents (user_id)
        ├── memory_entries (user_id)
        ├── user_profiles (user_id)
        └── skill_installs (user_id)

skills ── skill_installs (skill_id)
conversations ── messages (conversation_id)
conversations ── artifacts (conversation_id)
```

**一致性保证**：
- SQLite 默认启用外键约束（`PRAGMA foreign_keys = ON`）
- ON DELETE CASCADE：删除用户时级联删除其会话、消息、记忆等
- 应用层校验：在 API 层做业务逻辑校验（如"不能给不存在的会话发消息"）

**特殊处理**：Agent ID 在 messages 表中不是真正的外键（Agent 是注册表中的逻辑实体，不在数据库中），所以通过应用层保证一致性。

## Q16: 项目中如何使用数据库事务？有什么场景需要事务？

**参考答复：**

事务使用的几个场景：

**隐式事务**（SQLAlchemy 自动管理）：
- 单条 INSERT/UPDATE/DELETE 默认在自动提交模式下（autocommit=False 时在 flush 时生效）

**显式事务**（需要多个操作原子性时）：
```python
with db.begin():
    db.add(new_agent)
    db.add(new_skill_install)
    # 两条操作要么都成功，要么都回滚
```

**需要事务的场景**：
1. 创建 Agent + 初始化默认配置 → 原子操作
2. 安装 Skill + 更新安装计数 → 需要同时成功
3. 记忆提取 + 衰减更新 → 可选事务

当前项目中使用显式事务的场景不多，因为大部分操作是单表操作。随着业务复杂度的增加（如 Skill 市场引入"安装"操作），事务会变得更加重要。

## Q17: 孤儿的消息/会话数据是如何处理的？（如用户删除后，其会话是否级联删除？）

**参考答复：**

项目在 `app/main.py` 的 lifespan handler 中有一个**孤儿数据迁移**逻辑：

```python
# 将所有孤儿消息和会话迁移到第一个用户
first_user = db.query(User).first()
if first_user:
    orphan_conversations = db.execute(
        text("UPDATE conversations SET user_id = :uid WHERE user_id IS NULL"),
        {"uid": first_user.id}
    )
```

这个逻辑确保在系统初始化时，没有关联用户的数据被迁移到首个用户下。

**级联删除策略**：
- SET NULL / 迁移：对于可恢复的数据（会话、消息），不直接删除，而是迁移
- CASCADE：对于不可恢复的关联数据（如 skill_installs），用户删除后级联删除

这个策略的权衡是：保留数据可追溯性 vs 数据清洁度。

## Q18: chroma_db 和 SQLite 数据库之间的数据一致性如何保证？

**参考答复：**

ChromaDB（向量数据）和 SQLite（关系数据）的一致性是一个**弱一致性**设计：

**场景：RAG 文档管理**
1. 上传文档 → 文本分块存入 ChromaDB + 元数据存入 db（操作分开）
2. 删除文档 → 从 ChromaDB 删除向量 + 从 db 删除记录 + 删除文件 + 更新 mapping
3. 如果删除时 ChromaDB 操作成功但 db 操作失败 → 不一致

**场景：长期记忆**
1. 提取事实 → 写入 memory_entries 表（SQL） + 写入 ChromaDB collection（向量）
2. 如果 SQL 成功但 ChromaDB 失败 → 记忆有数据库记录但没有向量索引（无法被语义检索到）

**当前缓解措施**：
- 操作顺序：先写 SQL（关系数据），后写 ChromaDB（向量数据）
- 失败处理：ChromaDB 写入失败时记录 error 日志，后续可以通过重建索引修复

**理想方案**：使用事务性输出（Transactional Outbox）模式，但引入消息队列增加了部署复杂度。

## Q19: 数据库模型中的 JSON 字段（如 messages.content）在查询时如何使用？

**参考答复：**

SQLite 支持 JSON 函数来查询 JSON 字段：

```sql
-- 查询包含特定 @mention 的消息
SELECT * FROM messages
WHERE json_extract(content, '$.mentions') LIKE '%agent_id%'

-- 查询带附件的消息
SELECT * FROM messages
WHERE json_array_length(json_extract(content, '$.attachments')) > 0
```

在 SQLAlchemy 中使用原始 SQL：
```python
result = db.execute(text(
    "SELECT * FROM messages WHERE json_extract(content, '$.mentions') LIKE :mention"
), {"mention": f"%{agent_id}%"})
```

**限制**：
- SQLite 的 JSON 函数不能使用索引（除非建 JSON 索引的虚拟列）
- 复杂的 JSON 查询性能较差
- ORM 层面不支持 JSON 字段的类型安全操作

对于高频的 JSON 字段查询，建议将关键字段提升为独立列。

## Q20: 如果要引入数据版本控制（如 Agent 配置的历史版本），你会怎么设计？

**参考答复：**

Agent 配置版本控制的设计方案：

1. **新增 agent_versions 表**：
   ```sql
   CREATE TABLE agent_versions (
       id INTEGER PRIMARY KEY,
       agent_id INTEGER REFERENCES custom_agents(id),
       version_number INTEGER,
       config_snapshot TEXT,  -- JSON: 当时的完整配置
       change_description TEXT,
       created_by INTEGER REFERENCES users(id),
       created_at TIMESTAMP
   );
   ```

2. **写入逻辑**：每次 Agent 配置修改时，先插入 version 记录，再更新 Agent

3. **回滚机制**：用户可以选择一个历史版本恢复
   ```python
   def rollback_agent(agent_id, version_number):
       snapshot = get_version(agent_id, version_number)
       update_agent(agent_id, snapshot.config)
   ```

4. **diff 展示**：比较两个版本的 config_snapshot，展示配置变化

5. **保留策略**：只保留最近 N 个版本或最近 M 天的版本

这个设计类似 Git 的版本控制：每次修改产生一个 commit，可以随时回退到历史版本。对于 Agent 配置这种需要频繁实验的场景非常有用。

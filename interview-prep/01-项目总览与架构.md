# 面试准备：项目总览与架构

---

## Q0: 请简单介绍一下你的项目

**参考答复：**

AgentHub 是一个多 Agent 协作平台，核心目标是让用户通过自然语言与多个专业 AI Agent 协同完成复杂任务。项目采用 FastAPI + LangGraph + SQLite + ChromaDB 技术栈，后端约 15,000+ 行 Python 代码。

项目最核心的设计是一个 **智能编排器（Orchestrator）**，它充当中央调度大脑：接收用户消息后，自动进行复杂度分类（simple/moderate/complex），然后路由到合适的处理路径——简单问题直接回复，中等复杂度路由到专家 Agent，复杂任务则通过 LangGraph 动态规划引擎拆解为子任务并行执行。

架构上有几个亮点：
1. **多层记忆框架**：5 层记忆体系（指令记忆、短期记忆、工作记忆、摘要记忆、长期语义记忆），配合 LLM 驱动的记忆提取和 ChromaDB 语义检索
2. **多 LLM 后端统一抽象**：支持 DeepSeek、Tongyi/Qwen、OpenCode 三个 LLM 提供商，统一的 `chat/chat_stream` 接口
3. **双轨 Skill 系统**：工具类 Skill（可执行 Python 函数，封装为 LangChain Tool 支持 ReAct 循环）和能力类 Skill（Markdown prompt 注入）
4. **A-Tier 可配置 Agent**：每个 Agent 可独立配置记忆策略、规划模式、校验策略三个维度
5. **多通道通信**：WebSocket 实时流式 + SSE 降级 + HTTP 同步三通道

项目有完整的前端（单页应用）和后端，支持 JWT 认证、会话管理、Artifact 产出物管理，是一个生产级的多 Agent 协作平台。

## Q1: 项目的整体技术架构是怎样的？为什么选择这些技术？

**参考答复：**

技术架构上，选择了 **FastAPI** 作为 Web 框架，因为它原生支持 async/await，非常适合 LLM 调用这种 IO 密集型场景。**LangGraph** 用于 Agent 编排，它的 StateGraph 提供了状态机抽象，天然支持 checkpointer 做对话持久化，而且支持条件边实现动态路由。**SQLAlchemy + SQLite** 作为主数据库，轻量且零配置，适合中小规模部署。**ChromaDB** 做向量存储，配合 **DashScopeEmbeddings**（text-embedding-v4）做语义检索。

选择这套技术栈的核心理由是：全部 Python 生态，开发效率高；SQLite + ChromaDB 都是嵌入式数据库，部署简单无需额外服务；LangGraph 的条件边和 checkpointer 机制天然适合复杂 Agent 编排场景。

## Q2: 项目的核心设计模式是什么？Orchestrator 为什么设计成单例？

**参考答复：**

核心设计模式是 **中央编排器模式（Orchestrator Pattern）**。Orchestrator 作为全局单例，持有所有 Agent 注册表、LLM 后端池、工作流注册表、Skill 仓库和记忆服务。这个设计的优势是：所有路由决策集中在一处，避免分散的 if-else；Agent 注册和管理统一，支持热加载（用户创建的 Agent 即时生效）；记忆服务在整个生命周期内共享 ChromaDB 连接，避免重复初始化。

但单例模式也有代价：全局状态管理复杂，初始化顺序需要严格控制（先 LLM 后端 → 再 Skill → 再 Agent → 最后工作流和规划图），单元测试不够友好。实际项目中通过 FastAPI 的依赖注入来管理这个单例的生命周期。

## Q3: 你如何处理 LLM 调用的容错和降级？

**参考答复：**

项目有三层容错机制：

**第一层：启动时健康检查。** Orchestrator 初始化时对每个 LLM 后端做同步 health check（发 ping 消息，超时 8 秒），不可用的后端直接移除，避免运行时才发现问题。

**第二层：Agent 级降级链。** 每个 Agent 注册时配置了 `AGENT_FALLBACK_CHAIN`，例如 `opencode_coder` 的降级链是 `tongyi → deepseek`。当 Agent 调用失败时，自动切换到下一个可用后端重试。

**第三层：运行时异常捕获。** 所有 LLM 调用都包裹在 try/except 中，失败时返回明确的错误信息而非崩溃。工作流执行失败时降级为直接 LLM 调用。记忆服务初始化失败也不影响主流程（非致命错误）。

## Q4: 项目如何管理 prompt？为什么不用硬编码？

**参考答复：**

项目使用 **YAML 文件 + 变量替换** 的 prompt 管理体系。所有 prompt 模板集中存放在 `config/prompts/` 目录下，按功能分为 `orchestrator_prompts.yaml`（复杂度分类、路由 prompt）、`agent_prompts.yaml`（Planner、Summarizer、RAG 等 prompt）、`workflow_prompts.yaml`（计划评估、记忆压缩、任务执行等）。

通过 `PromptLoader` 加载，支持 `{variable}` 占位符替换。这样做的好处是：prompt 与代码解耦，调优 prompt 不需要改代码；所有 prompt 集中管理，方便做 A/B 测试；YAML 格式可读性好，非技术人员也能参与 prompt 优化。

## Q5: 项目中的多 Agent 通信是如何实现的？

**参考答复：**

多 Agent 通信有两种模式：

**1. @mention 模式：** 用户在消息中使用 `@agent_id` 来显式指定某个 Agent。支持单 Agent mention 和多 Agent 群聊（多个 @mention 同时触发）。群聊时 Orchestrator 将消息并行发送给多个 Agent，收集所有回复后汇总返回。

**2. 动态规划模式：** 对于复杂任务，PlannerAgent 将任务拆解为子任务列表，然后通过 LangGraph 并行调度多个子 Agent 执行，最后由 SummarizerAgent 汇总结果。这种模式下 Agent 间的通信是通过 GraphState 共享状态实现的，不需要 Agent 之间直接对话。

**3. Agent Builder 模式：** 专门的 `agent_builder` Agent 负责响应用户的 Agent 管理请求（创建、修改、删除 Agent），它可以直接调用 `manage_agent` 工具完成数据库操作。

## Q6: 为什么选择 LangGraph 而不是其他 Agent 框架？

**参考答复：**

选择 LangGraph 有几个关键原因：

1. **状态机抽象自然**：LangGraph 的 StateGraph 用节点（处理步骤）和边（流转逻辑）来描述 Agent 流程，这比手写 if-else 路由更清晰且可维护
2. **条件边支持动态路由**：`add_conditional_edges` 可以根据运行时状态决定下一步走向（如评估结果后决定是重新执行还是生成摘要），这是纯 chain 模式做不到的
3. **内置 checkpointer**：通过 `AsyncSqliteSaver` 自动持久化对话状态到 SQLite，实现对话记忆的跨请求持久化，不需要自己实现
4. **与 LangChain 生态兼容**：LangChain 的 Tool 抽象、ReAct executor 可以直接集成到 LangGraph 节点中
5. **相对轻量**：相比 AutoGen/CrewAI 等框架，LangGraph 更底层、更灵活，不会被框架的设计理念限制

## Q7: 项目支持哪些 LLM 模型？为什么选这些？

**参考答复：**

项目通过三个后端接入了多款模型：

- **Tongyi/Qwen 系列**（通过 DashScope API）：`qwen-plus`、`qwen-long`。选它作为默认后端，因为中文能力强、性价比高
- **DeepSeek 系列**（通过 DeepSeek API）：`deepseek-chat`、`deepseek-coder`。代码能力出色，用作代码审查 Agent
- **OpenCode Zen**（免费路由）：`deepseek-v4-flash-free`、`big-pickle` 等 7 款免费模型。提供零成本的免费档选项

选择多后端的理由是：不同模型有不同的优势领域（Qwen 中文强、DeepSeek 代码强），通过统一的后端抽象层可以实现"取长补短"，而且多后端提供了自然的容灾降级能力。

## Q8: 项目的 WebSocket 流式输出是如何实现的？

**参考答复：**

WebSocket 流式输出的核心是一个 **异步队列（asyncio.Queue）** 的生产者-消费者模式：

- **生产者**：`get_chat_response()` 在路由处理过程中，将中间结果、token 块、artifact 等推入 `progressive_queue`
- **消费者**：WebSocket handler 从队列中消费事件，按类型分别处理：token 事件实时推送打字机效果、intermediate 消息展示子 Agent 执行进度、artifact 事件推送代码/HTML 产出物、thinking 事件更新前端状态指示器

SSE 降级：当 WebSocket 不可用时，`get_chat_stream()` 通过 async generator + `StreamingResponse` 提供 HTTP SSE 流式输出，将完整回复按 `TOKEN_CHUNK_SIZE=8` 分块 + `TOKEN_DELAY_MS=15ms` 延迟模拟打字机效果。

## Q9: 项目如何处理并发请求？

**参考答复：**

并发处理主要依赖 FastAPI 的异步特性：

1. **async/await 全链路**：从路由 handler 到 LLM 调用（httpx async client）到数据库操作（aiosqlite），全部是异步非阻塞的
2. **每个请求独立的队列**：`get_chat_stream()` 中每个请求创建独立的 `asyncio.Queue`，避免流式输出的并发干扰
3. **数据库连接管理**：通过 SQLAlchemy 的 `SessionLocal` 工厂模式，每个请求获取独立 session，使用完关闭
4. **ChromaDB 客户端**：使用 PersistentClient（文件级锁），读取操作天然并发安全
5. **LangGraph checkpointer**：AsyncSqliteSaver 基于 aiosqlite，支持异步并发读写

当前架构的瓶颈在 LLM API 调用的并发限制（API 提供商的 rate limit），而非服务端本身。

## Q10: 项目中最大的技术挑战是什么？如何解决的？

**参考答复：**

最大的技术挑战是 **复杂任务的自动拆解和调度**。

问题本质是：用户的自然语言请求可能是模糊的、多步骤的（如"帮我开发一个用户管理系统"），传统的关键词匹配无法准确判断这是一个需要拆解的任务，更无法生成合理的执行计划。

解决方案分三步：
1. **LLM 复杂度分类**：用 LLM 将用户输入分类为 simple/moderate/complex/agent_management，比关键词匹配更准确，且有关键词降级兜底
2. **PlannerAgent 任务拆解**：对于 complex 任务，PlannerAgent 生成包含子任务列表、依赖关系、分配 Agent 的 JSON 计划
3. **LangGraph 动态执行**：通过 execute_tasks → evaluate_results → generate_summary 的状态图，支持并行执行子任务、评估结果后重规划、最终汇总

这里面最 tricky 的是 prompt 工程：PlannerAgent 的 prompt 需要精确控制输出格式（严格 JSON），同时要让 LLM 理解每个 Agent 的能力边界从而合理分配任务。

## Q11: 介绍一下项目的部署架构

**参考答复：**

项目目前设计为**单机部署**模式：

- FastAPI 应用通过 uvicorn 启动，监听单端口
- SQLite 作为主数据库（`agenthub.db`），LangGraph checkpointer 使用独立的 SQLite（`agenthub_memory.sqlite`）
- ChromaDB 持久化到本地 `chroma_db/` 目录
- 前端静态文件通过 FastAPI 的 `StaticFiles` 中间件挂载
- JWT 鉴权，密钥通过环境变量 `.env` 配置
- 文档上传存储在 `data/` 目录

这个架构的优势是部署简单（零外部依赖），适合小团队或个人使用。如果要扩展为多实例部署，需要将 SQLite 替换为 PostgreSQL，ChromaDB 替换为服务化部署，并引入 Redis 做 WebSocket 的 pub/sub。

## Q12: 项目的测试策略是怎样的？

**参考答复：**

项目使用 pytest 框架，测试配置在 `pytest.ini` 中定义。测试策略分几个层面：

1. **单元测试**：对核心逻辑（如复杂度分类的降级规则、token 分段算法）做独立测试
2. **集成测试**：测试 API 路由的端到端行为，使用 FastAPI 的 TestClient
3. **LLM 调用 Mock**：对于涉及 LLM 调用的测试，使用 mock 避免实际 API 消耗

当前测试覆盖主要集中在工具函数和 API 路由层面。由于 LLM 调用的非确定性，核心编排逻辑的测试更多依赖集成测试而非纯单元测试。

## Q13: 项目如何确保代码质量？

**参考答复：**

代码质量保障措施：

1. **类型注解**：Python 代码全面使用 type hints，提高可读性和 IDE 支持
2. **结构化日志**：统一的 logger 模块，关键路径有详细日志，便于问题排查
3. **配置集中管理**：所有魔法数字和硬编码常量集中在 `config.py` 中，支持环境变量覆盖
4. **错误处理规范**：LLM 调用、数据库操作、文件 IO 等易出错的操作都有 try/except + 日志
5. **响应完整性检查**：`check_response_completeness()` 函数在 LLM 返回后检查回答是否被截断
6. **初始化顺序明确**：Orchestrator 的 `__init__` 中有清晰的注释标注初始化顺序（先核心后业务），避免依赖问题

## Q14: 如果要你重新设计这个项目，你会做哪些改变？

**参考答复：**

如果重新设计，几个改进方向：

1. **拆分 Orchestrator 单体**：当前 `orchestrator.py` 约 3256 行，职责过重。会拆分为 Router（路由决策）、AgentRegistry（Agent 管理）、SkillManager（Skill 管理）、MemoryCoordinator（记忆编排）等独立模块
2. **引入依赖注入框架**：用 `dependency-injector` 或 FastAPI 的 Depends 更规范地管理依赖，替代当前的手动单例管理
3. **Agent 间通信协议标准化**：当前 Agent 间通过 GraphState 字典传递信息，缺少类型约束。会引入 Pydantic 模型定义 Agent 间通信协议
4. **引入消息队列**：对于复杂任务的子任务执行，用消息队列解耦而非内存中并行，提高可靠性和可扩展性
5. **可观测性**：引入 OpenTelemetry 做分布式追踪，特别是多 Agent 协作场景下的调用链追踪

## Q16: AgentHub 和 LangChain 的 AgentExecutor 有什么区别？

**参考答复：**

LangChain 的 AgentExecutor 是一个通用的 ReAct 循环执行器：接收 prompt + tools，循环执行"思考 → 选择工具 → 执行工具 → 观察结果"直到产生最终答案。

AgentHub 在 LangChain 之上做了多层增强：

1. **智能路由前置**：在进入 AgentExecutor 之前，先经过复杂度分类和路由决策，简单问题直接回复不浪费 LLM 调用
2. **多 Agent 协作**：不是单个 Agent + tools，而是多个专业 Agent 的协作编排
3. **记忆框架**：5 层记忆在 LLM 调用前后自动增强上下文和提取新记忆
4. **产物管理**：支持 Artifact（代码、HTML 预览、Markdown 报告）的结构化输出和持久化
5. **A-Tier 配置**：每个 Agent 独立配置记忆/规划/校验策略，比 LangChain 的全局配置更灵活

## Q17: 前端和后端是如何协作的？API 契约是怎样的？

**参考答复：**

前后端分离架构，前端是独立的单页应用，后端通过 REST API + WebSocket 提供服务。

API 契约按功能域划分：
- `/api/chat` — 聊天核心接口（POST 同步 + SSE 流式）
- `/ws/{conversation_id}` — WebSocket 实时通信
- `/api/missions` — 会话 CRUD
- `/api/agents` — Agent 管理
- `/api/skills` — Skill 市场
- `/api/knowledge` — 知识库/RAG 管理
- `/api/memory` — 记忆管理

前后端通过 JWT token 鉴权。WebSocket 连接也在 URL 参数中携带 token 做认证。前端通过 `agent_override` 参数实现"切换 Agent 后发消息自动 @mention"的体验优化。

## Q18: 项目中的 artifacts（产物）是如何管理的？

**参考答复：**

Artifact 系统用于管理 Agent 产出的结构化内容（代码文件、HTML 预览、Markdown 报告等）。

**数据模型**：`artifacts` 表包含 id、conversation_id、agent_id、type（code/html_preview/markdown）、title、content、language 字段。

**生成方式**：Agent 在流式输出中将其包装为特殊标记（如 `\`\`\`artifact`），前端解析后渲染为独立卡片。WebSocket 和 SSE 通道中，artifact 作为独立事件类型推送。

**持久化**：Artifact 写入数据库，关联到会话，用户可以回看历史会话中的所有产物。

## Q19: 项目的限流和防护策略是怎样的？

**参考答复：**

当前项目的防护策略主要包括：

1. **JWT 鉴权**：所有 API 和 WebSocket 都需要有效 token
2. **密码加密**：使用 bcrypt（passlib）做密码哈希
3. **Agent 执行限制**：ReAct 循环最大迭代次数 3，规划模式最大迭代 10，防止无限循环消耗 token
4. **输入长度**：通过 LLM 的上下文窗口自然限制（超过的通过记忆策略压缩）

目前没有实现基于用户/IP 的 rate limiting，这是后续需要加强的部分。对于生产部署，应该在 API Gateway 层（如 Nginx）或 FastAPI 中间件层添加限流。

## Q20: 这个项目给你最大的收获是什么？

**参考答复：**

最大的收获是对 **Agent 系统复杂性的深刻理解**。

具体来说：
1. **编排 > 模型**：一个好的编排/路由系统比选一个更强的模型更重要。项目中大部分"智能"来自 Orchestrator 的路由决策和 prompt 工程设计，而非模型本身
2. **记忆是 Agent 系统的核心差异化**：没有记忆的 Agent 只能做单轮问答，有了多层记忆才能真正理解用户、跨会话积累知识
3. **容错设计是生产级的标志**：LLM 服务不稳定是常态，多层降级、健康检查、优雅降级这些"脏活"才是系统可靠性的保障
4. **LangGraph 是一个强大的抽象**：状态图模型天然适合描述 Agent 的执行流程，特别是条件边和 checkpointer 的组合
